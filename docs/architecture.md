# TaaSim Urban Mobility Platform — Architecture & File Guide

**Build the data infrastructure that moves Casablanca.**

This repository contains a production-grade streaming data platform for urban mobility analysis. The codebase implements a **Kappa Architecture** with Apache Kafka, Flink, and Cassandra, plus batch analytics via Apache Spark.

---

## 📁 Project Structure

### **Core Streaming Engines (Flink Real-Time Jobs)**
- **`job1_gps_normalizer.py`** — **Flink Job 1: GPS Normalizer**
  - Consumes `raw.gps` from Kafka
  - Validates coordinates, assigns event-time watermarks (3-min lateness)
  - Map-matches raw GPS to 16 Casablanca zones
  - Anonymizes coordinates (replaces with zone centroids)
  - Sinks to Cassandra (`vehicle_positions`) + Kafka (`processed.gps`)

- **`job2_demand_aggregator.py`** — **Flink Job 2: Demand Aggregator**
  - Consumes `processed.gps` (vehicles) + `raw.trips` (requests)
  - 30-second tumbling windows per zone (event-time based)
  - Counts distinct active vehicles + pending requests
  - Computes supply/demand ratio
  - Sinks to Cassandra (`demand_zones`) + Kafka (`processed.demand`)

---

### **Data Producers (Streaming Simulation)**
- **`ProducerGps.py`** — GPS Telemetry Stream
  - Replays Porto taxi trajectories at 10× accelerated speed
  - Adds realistic noise (±20m GPS drift, 5% blackout probability)
  - Publishes JSON to Kafka topic `raw.gps`
  - Output: `{taxi_id, timestamp, latitude, longitude, speed, status}`

- **`ProducerTrips.py`** — Trip Reservation Stream
  - Generates on-demand trip requests following Porto's demand curve
  - Peak hours: 7-9am, 5-7pm (3-5× normal rate)
  - Friday 12-2pm reduced rate (0.5×)
  - Publishes to Kafka topic `raw.trips`
  - Output: `{trip_id, rider_id, origin_zone, destination_zone, requested_at, call_type}`

---

### **Batch Processing & ML**
- **`DataClean.ipynb`** — Spark ETL Pipeline
  - Loads Porto taxi CSV (1.7M trips) from `train.csv`
  - Applies geospatial transformation: Porto → Casablanca bounding box
  - Cleans NYC TLC parquet data (3 months, ~10M rows)
  - Removes outliers, validates data quality
  - Exports clean datasets for ML training
  - Output: `casablanca_teleported.csv`, `newyork/processed/nyc_clean.parquet`

- **`spark_porto_casa.ipynb`** — Feature Engineering & Analytics
  - Spark SQL jobs for batch KPI computation
  - Prepares feature matrix for ML model training

### **Data & Configuration**
- **`train.csv`** — Porto Taxi Trajectories (ECML 2015)
  - 1.7 million completed taxi trips (July 2013 – June 2014)
  - Columns: TRIP_ID, CALL_TYPE, TAXI_ID, TIMESTAMP, POLYLINE (GPS coordinates), DAY_TYPE
  - Source: https://www.kaggle.com/c/pkdd-15-predict-taxi-service-trajectory-i

- **`newyork/`** — NYC TLC Data (Batch Reference)
  - 3 months of NYC taxi trips in Parquet format
  - Used for demand pattern comparison + ML baseline
  - Source: https://www.nyc.gov/site/tlc/about/tlc-trip-record-data.page

- **`casablanca_teleported.csv`** — Transformed GPS Data
  - Output from `DataClean.ipynb`
  - Porto trajectories remapped to Casablanca coordinates
  - Ready for Kafka producer ingestion
  - Format: TRIP_ID, TAXI_ID, TIMESTAMP, CASA_POLYLINE (JSON array)

- **`cassandra_schema.cql`** — Cassandra DDL
  - Creates keyspace `taasim` with 3 tables:
    - `vehicle_positions`: Current taxi locations (partition: city, zone_id)
    - `trips`: Trip history (partition: city, date_bucket)
    - `demand_zones`: Aggregated demand per zone per time window (partition: city, zone_id)

- **`docker-compose.yaml`** — Infrastructure Stack
  - Single-machine Docker setup for all services:
    - Apache Kafka (KRaft mode, 1 broker)
    - Apache Cassandra (4.1, single node)
    - Apache Flink (1 Job Manager + 1 Task Manager)
    - Apache Spark (master + 1 worker)
    - MinIO (S3-compatible object store)
    - Grafana (dashboards + visualization)

## 📝 Key Concepts

**Event-Time Processing:**
- All Flink jobs use event-time (not processing-time) to handle out-of-order GPS and late trip requests
- Watermark strategy: 3-minute allowed lateness (configured globally)
- Ensures correct demand aggregation despite network delays

**Geospatial Anonymization:**
- Raw GPS coordinates replaced with zone centroids before persistence
- 16 zones in Casablanca (arrondissements + commercial areas)
- Raw coordinates NEVER stored (privacy by design)

**Kappa Architecture:**
- Single unified pipeline: Kafka is the system of record
- Historical data replayed through same topics as live events
- Spark handles offline analytics (separate concern)
- No Lambda complexity: one processing engine (Flink) handles both real-time and replay
